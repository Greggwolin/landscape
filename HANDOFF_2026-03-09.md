### Issue 1a — Extraction Field Flattening (NOT STARTED)

**Problem:** After successful extraction, Landscaper tells the user "I was unable to extract" or "could only extract 1 field" — even though 43 units were extracted at ~0.90 confidence. This is a presentation bug, not an extraction bug.

**Root cause:** Batched extraction stores all unit-level data under a single `field_key = 'unit_number'` with each row containing a full JSON blob in `extracted_value`:
```json
{"unit_number": "101-A", "unit_bedrooms": 1, "unit_bathrooms": 1, "unit_unit_type": "1/1.00", "occupancy_status": "Occupied", "unit_current_rent": 1825}
```

The `<ingestion_state>` block injected into Landscaper's system prompt (`ai_handler.py` → `_get_ingestion_context()`) shows only **2 unique field_keys** (unit_count + unit_number), so Landscaper interprets this as near-total extraction failure.

**Key files:**
- `backend/apps/knowledge/services/extraction_service.py` — `EXTRACTION_BATCHES`, `ChunkedRentRollExtractor`, `RENT_ROLL_CONFIG`
- `backend/apps/landscaper/ai_handler.py` lines 1525-1650 — `_get_ingestion_context()` builds the `<ingestion_state>` block

**DB state (project 122, `ai_extraction_staging`):**
- 53 total rows: 46 pending, 7 rejected
- 43 rows with `field_key='unit_number'`, scope `unit` — each containing a full data blob
- 3 rows with `field_key='unit_count'`, scope `unit_type`
- 7 rejected: city, income_parking, property_name, state, street_address, total_units, zip_code

**Possible fix directions:**
- **(A) Flatten at extraction time:** Change `ChunkedRentRollExtractor` to store each unit's fields as separate staging rows (e.g., `field_key='unit_bedrooms'` per unit). More correct but big change.
- **(B) Fix ingestion_state reporting:** Change `_get_ingestion_context()` to detect array-type `field_key='unit_number'` blobs and report them as "43 units extracted with 6 fields each" instead of "2 unique field_keys." Smaller, more targeted fix.
- **(C) Hybrid:** Keep storage as-is but add a unit-count summary line to the ingestion_state context and adjust the system prompt so Landscaper knows unit_number blobs = successful unit extraction.

**Recommendation:** Option B or C — the extraction data is actually fine, just the presentation to Landscaper is misleading.
