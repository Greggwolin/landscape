The land development product, end to end

One page. Six stages, each already de-risked by a working prototype. Below them, the eight decisions that lock the design before anything goes to the coding side.

Design rule: land is the only property type this product knows. Every screen gets to be exactly what land work needs — no compromises borrowed from income property. The engines, database, documents, knowledge, and map machinery underneath are shared Landscape infrastructure, untouched.

The spine — six stages

0

Find & create

A workspace map that belongs to the app, not to any project. A new deal starts here: search or click county tax parcels, or draw a boundary — and that act creates the project, with geometry, acreage, and parcel numbers already filled in before the first document drops.

Grounded in: your live map engine + the boundary/overlay tools it already has · new scope, existing machinery
1

Intake

Drop what exists — one colored exhibit or a full entitlement folder. The system reads what it can, tags every value as read / assumed / your call, respects document precedence (recorded beats approved beats proposed), tracks entitlement clocks as risk items, and is honest about scanned files it can't read. Judgment calls queue in a decision docket: blockers versus pre-answered defaults, each with in-tile Q&A.

Proven in: prototypes 3–6 (decision tiles, docket at scale, Peoria exhibit read, Red Valley entitlement chain)
2

Structure & parcel table

The Area → Phase → Parcel spine gets decoded from the documents and confirmed by you — never silently assumed. Then the parcel table is authored: on the map by multi-select or paint, or row by row in the table, with the canonical land use library (families → types → products, customizable) validating every entry and defaulting densities. Grouped and flat table views stay in sync.

Proven in: prototypes 5b–5i, 8 (taxonomy confirm, toggleable table, paint/select authoring)
3

Site systems & quantities

Streets, tracts, and channels become first-class map objects with their own spec panels — class from a standards library, lengths measured from the plans, widths from standard sections, quantities derived per segment. Everything tagged by method, everything reconciled line-by-line against the engineer's tabulations when they arrive. Native CAD/GIS deliverables, when requested up front, make this exact instead of estimated.

Proven in: prototypes 9, 10, 10b (derivation worksheet, clickable streets, class symbology)
4

Economics

Quantities meet unit costs to become phase budgets; products meet lot pricing to become revenue; parcels get takedown dates that roll up into schedule tiles linked both ways to the map. Change one parcel's date or one street's class and everything downstream re-derives.

Interaction proven in: prototype 7 (takedown tiles) · engines already live in Landscape
5

Analysis & outputs

Cash flows, returns, sensitivities — the calculation engines you already trust — plus the standard and custom output artifacts, every number tracing back through the chain: output → assumption → decision → document. This stage is deliberately last to design; it inherits everything above it.

Engines: existing · output workflow: the second design exercise, after intake ships

Decision register — your calls, before anything gets built

What happens after the register is locked

Decision-first, then handoff. Each locked decision flows into a phased implementation spec with explicit human gates — the coding side gets instructions, not open questions. First slice ships to you, you run it on the pilot project, and the next slice is chosen from what the real workflow demands. No stage gets gold-plated in the dark.
