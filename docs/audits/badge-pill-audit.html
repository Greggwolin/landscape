Consistency Matrix
DimensionAssessmentDetail
Property type colors Centralized Single source of truth in tokens.css. All consumers resolve via getPropertyTypeTokenRef(). 50+ aliases mapped.
Lifecycle stage colors Consistent STAGE_COLORS object shared between ActivityFilter and CategoryTreeItem. Maps to CoreUI semantic names.
Semantic chip tokens Centralized 8 intents defined in tokens.css with light/dark variants. All use CSS variable pattern: --chip-{intent}-{bg|border|text|outline}.
Status color mapping Consistent success=green, warning=amber, danger=red, info=blue, secondary=gray. Uniform across StatusChip, workbench, tab badges, version badges.
Border radius 3 conventions 4px (status badges, studio, lifecycle), 6px (budget-mode), 999px (property type pills, channel pills, extraction filters, semantic chips). No documented rule for when to use which.
Implementation approach Fragmented 4 methods: CoreUI CBadge, custom React wrappers, CSS-only classes, CSS-in-JS inline. StatusChip and SemanticBadge both wrap CBadge but expose different APIs.
Growth/ops badge colors Hardcoded GrowthBadge uses hardcoded hex (#f6ad55, #68d391) instead of CoreUI variables or tokens. Works but bypasses theme system.
Naming convention Inconsistent "Badge" vs "Chip" vs "Pill" used interchangeably. StatusChip is a badge. MediaBadgeChips renders badges. ExtractionFilterPills are toggles.
Dark mode Mostly covered CSS-variable components auto-adapt. Semantic chips have explicit dark overrides in tokens.css. GrowthBadge hardcoded colors need verification.
Accessibility Gaps Color-only differentiation in MediaBadgeChips and lifecycle letter badges. Most status badges lack role="status" or aria-label.

Key Findings

1
Three well-governed color systems. Property type tokens (7 types, 50+ aliases), lifecycle stages (6 stages via STAGE_COLORS), and semantic category chips (8 intents with light/dark) are all centralized and consistent. These are the models to follow.
2
GrowthBadge bypasses the theme. Hardcoded #f6ad55 and #68d391 instead of CoreUI variables. If the theme ever changes, these badges won't follow. Low risk but easy to fix.
3
Border radius has no rule. 4px, 6px, and 999px all appear for status indicators. Suggest: 999px = identity pills (property type, channel), 4px = status/state badges, 6px reserved for interactive mode selectors.
4
Two competing CBadge wrappers. StatusChip (status enum API) and SemanticBadge (intent routing API) both wrap CBadge. Could unify under SemanticBadge with a "status" intent.
5
Badge/Chip/Pill naming is ad-hoc. No semantic distinction. StatusChip renders badges. MediaBadgeChips renders chips. ExtractionFilterPills are toggles. A naming convention would help onboarding.
6
9 CSS-only patterns lack React wrappers. No TypeScript safety net if classes are renamed. Low priority — most are stable and domain-specific.
7
Accessibility gaps. Lifecycle letter badges and MediaBadgeChips rely on color alone. No role="status" on state-communicating badges. VersionBadge is the only one with proper button semantics.
8
No dead badge code detected. All 22 patterns have active consumers. No orphaned components.
